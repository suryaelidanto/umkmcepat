## Status History

- 2026-07-08: Proposed after discussing a user-facing Agentation-style visual feedback mode for UMKM Cepat generated previews.

## Problem Statement

After UMKM Cepat generates a website, users can inspect the preview and continue chatting with AI, but describing visual changes through text alone is awkward. A non-technical UMKM user may want to say “this headline is too big,” “change this button text,” or “this section should feel cleaner,” while pointing at the exact part of the generated website. Today they must describe the element manually and hope the AI understands which part they mean.

This creates friction in the most important iteration loop: preview -> feedback -> AI edit -> rebuild. The product has a generated app source/edit path, but the UX still asks users to translate visual feedback into written instructions. That is especially bad for UMKM users because they should not see or understand CSS selectors, React components, source paths, DOM tags, or generated source structure. They should be able to click what they see, leave a natural note, collect a few comments, and ask AI to revise the website.

Agentation is strong prior art for this problem. Its user-facing workflow is: activate a visual feedback tool, hover elements to see highlight, click an element or selected text, add a note, and output structured context that an AI coding agent can use. UMKM Cepat should adapt that feel to its own product: not as a developer tool that copies markdown into Claude Code, but as a user-facing annotation mode inside the project workspace that attaches comments to chat/edit requests and reuses the platform's safe generated source edit/build loop.

## Solution

UMKM Cepat will add a visual annotation mode for generated previews. The mode is available only when a project has a visible generated preview: either the latest successful preview or the last good preview when the newest attempt failed. It is not available before the first build.

The activation entry point lives in the workspace preview top bar near the existing preview/code controls, not as a floating button inside the generated website. This avoids collision with buttons or floating UI generated by the AI website itself. When the user activates annotation mode, the left chat panel temporarily becomes a simple “Visual comments” panel. The preview enters annotation mode: hovering over elements highlights them, clicking an element opens a small note popup near the element, selecting text and then annotating captures the selected text, and saved comments appear as numbered pins in the preview and as a list in the left panel.

The user can add multiple comments before sending. The left panel shows each comment with a friendly label, not technical internals. Labels combine a semantic guess and visible text, such as `Judul utama — "Servis motor rapi tanpa tebak-tebakan"`, `Tombol — "Pesan via WhatsApp"`, or `Kartu layanan — "Cuci setrika"`. Hidden metadata captures enough DOM context for the edit agent, including selector/path, tag, visible text, selected text, bounding box, nearby text, and CSS classes. The UI never shows selectors, tags, source paths, component names, or computed styles to the user.

When the user clicks “Send to AI,” UMKM Cepat creates a chat-history entry summarizing the visual comments in Indonesian, attaches the hidden annotation payload to the edit instruction, reuses the existing generated source edit route where possible, and runs the safe edit/build path. If the edit/build succeeds, annotations are cleared. If the edit/build fails, annotations remain so the user does not lose feedback. The latest successful preview remains safe according to the existing edit/build preservation rules.

This PRD intentionally scopes the first version as a practical MVP. It should feel like Agentation in the core interaction: activate, hover highlight, click, comment, pin, list, send. It should not copy Agentation's full developer tooling, MCP integration, webhooks, source-file mapping, React component tree inspection, computed-style forensics, area selection, multi-element selection, animation pause, or external copy-to-markdown workflow.

## Implementation Decisions

- The feature is user-facing for UMKM users, not an internal developer QA tool. All visible copy should use friendly Indonesian and avoid developer terms.
- The annotation activation control should live in the workspace preview top bar near the preview/code controls. It should not be a floating button inside the generated preview because generated websites may already contain floating buttons or chat widgets.
- Annotation mode is a mode on top of the preview, not a third content tab. The preview remains visible while annotation mode changes how hover and click behave.
- Annotation mode should be available only when there is a generated preview or last good preview. It should be disabled or hidden before the first successful build.
- When annotation mode is active, the left workspace panel should temporarily switch from normal chat to a visual comments panel. This panel should be simple: title, short instructions, current comment list, optional general instruction textarea, and send button.
- The normal chat history should be hidden while annotation mode is active to reduce noise. Users can exit annotation mode to return to chat.
- The annotation comments panel should remain product-simple. It should not expose developer inspector details.
- Each saved annotation should render as a numbered pin in the preview and as a list item in the left comments panel.
- Hovering a list item should highlight its target element in the preview when possible. Hovering a pin should reveal or emphasize the related comment.
- Clicking an element in annotation mode should open a small popup near the clicked element. The popup should contain a textarea and actions to add or cancel the comment.
- The popup plus list pattern is intentionally hybrid: quick local input near the element, organized review in the left panel.
- Text selection is part of MVP. If selected text exists when the user annotates, it should be captured as `selectedText` and shown in the friendly label.
- Area rectangle selection is out of MVP. It is ambiguous for source mapping and not necessary for the first UMKM-facing flow.
- Multi-element selection is out of MVP. A user should create multiple separate comments instead.
- The annotation target label should combine semantic guess and visible content, for example `Judul utama — "..."`, `Tombol — "..."`, `Kartu layanan — "..."`, `Gambar — "..."`, or `Bagian — "..."`.
- The annotation target selection should prefer meaningful elements over tiny nested internals. If the user clicks text inside a button, target the button. If the user clicks inside a link, target the link. If the user clicks inside a card, target the closest meaningful card/section when appropriate. If the user clicks a heading, target the heading.
- Agentation's element selection mechanics should be adapted conceptually:
  - Use coordinate-based element lookup (`elementFromPoint`) inside the preview document.
  - Use capture-phase click listeners so annotation mode intercepts before generated links/buttons navigate.
  - Ignore clicks inside annotation UI.
  - Block normal interactive behavior while annotation mode is active.
  - Use hover listeners to calculate bounding boxes and render highlight overlays.
  - Use `window.getSelection()` to capture selected text.
  - Use heuristics to identify button/link/heading/paragraph/image/container labels.
- Because UMKM Cepat previews are iframes, the annotation logic should run in the preview document and communicate with the parent workspace through `postMessage`. The parent workspace should own annotation state and UI.
- The private preview route or runtime proxy should inject a lightweight annotation bridge script into preview HTML only for private workspace previews. Published public pages should not include annotation instrumentation.
- The parent workspace should send messages to the preview iframe to turn annotation mode on/off, request highlight changes, and clear markers when needed.
- The preview iframe should send messages to the parent for hover target, clicked target, selected text, and annotation marker geometry.
- The initial annotation payload should include a hidden technical target and a friendly label:

```ts
// Decision shape, not final implementation code
{
  id: string;
  label: string;
  comment: string;
  selectedText?: string;
  target: {
    tag: string;
    text: string;
    selectorPath: string;
    boundingBox: { x: number; y: number; width: number; height: number };
    nearbyText?: string;
    classes?: string;
  };
}
```

- The user-facing list should show only `label`, selected text when relevant, and the comment. The hidden target goes to AI/edit logic only.
- Payload values must be capped. Recommended caps: selected text 500 characters, element text 300 characters, nearby text 500 characters, classes 300 characters, comment 1,000 characters, total annotations 20 per send.
- Sending visual comments should create a user-visible chat summary. The summary should be friendly and readable, such as `Aku kirim 3 komentar visual untuk direvisi` followed by bullet points with labels and comments.
- Sending visual comments should also submit hidden annotation context to the edit flow. The edit instruction should include the general instruction textarea plus structured annotation context.
- The MVP should reuse the existing generated source edit/build route if possible. This avoids adding a new route before the product proves the interaction.
- The edit agent should receive annotation context in a way that encourages targeted changes. It should know visible label, selected text, comment, DOM selector/path, and nearby text.
- The existing edit route currently accepts instruction/commands. The first implementation may serialize annotations into the instruction string, but a typed annotation field is preferred if it can be added without broad route churn.
- The edit/build flow after visual comments should behave like other post-preview edits: create a new snapshot/build, preserve latest successful preview if the new build fails, and surface build progress.
- On successful edit/build, clear current annotations and exit annotation mode.
- On failed edit/build, keep annotations and show a recoverable message so the user can retry or adjust comments.
- The comments list should support deleting annotations before send. Editing existing annotation text is desirable in MVP if cheap, but can be implemented as delete-and-readd if necessary.
- The generated website should not permanently include annotation UI. Annotation overlays/pins are workspace UI/instrumentation only.
- The feature should not introduce arbitrary script execution beyond the controlled private preview bridge. Generated source remains untrusted data and must not be imported into the Next.js control plane.
- The annotation bridge must avoid interfering with preview-ready messaging. Existing `umkmcepat-preview-ready` behavior remains unchanged.
- The feature should not require adding Agentation as a dependency. Agentation is research/reference material. UMKM Cepat should implement its own minimal bridge adapted to the iframe/runtime architecture.
- The feature should not copy Agentation's license-restricted code directly. Use the observed concepts and implement cleanly for UMKM Cepat.
- The UI should stay minimal. Avoid toolbar clutter, floating devtool chrome, raw markdown output, copy-to-clipboard workflows, or MCP concepts.

## Testing Decisions

- Highest-value seam: project workspace behavior around annotation mode and post-preview edit. Tests should prefer workspace state and route behavior over private DOM helper implementation.
- Add unit tests for workspace state transitions: annotation mode is available only with preview/last-good preview, normal composer hides while annotation mode is active, annotation panel appears, and completed-build review state is not overwritten by stale cards.
- Add unit tests for annotation payload formatting: friendly labels, selected text capture, comment caps, hidden target serialization, and chat summary generation.
- Add unit tests for target label heuristics where feasible: headings become `Judul utama` or `Judul`, buttons/links become `Tombol`, paragraphs become `Teks`, images become `Gambar`, sections/cards become `Bagian` or `Kartu`.
- Add route or handler tests for visual comment submission to the edit flow. The test should assert that annotation summary is added to chat-facing data and annotation context reaches edit/build instruction.
- Add edit-route tests showing annotation-driven edits preserve latest successful preview when build fails, following existing failed edit preservation prior art.
- Add preview/proxy tests for annotation bridge injection only if implemented at the route/proxy seam. Assert private preview HTML gets the bridge when requested and public/published routes do not.
- Add browser-level or component-level tests later for hover/click behavior if a reliable test harness exists. Do not over-invest before MVP UX is validated manually.
- Existing prior art in the codebase includes workspace sync tests, brief-flow tests, edit route tests, runtime/preview route tests, source route tests, generated source tests, agent tool runner tests, and build worker tests. New tests should follow those patterns.
- Good tests should assert external behavior and contracts: which composer/panel appears, what payload is produced, what route receives, what happens on success/failure. Avoid brittle tests of internal hover implementation details.
- Manual QA is required for the first slice because the core value is visual feel: hover highlight, popup placement, pins, sidebar list, and iframe interaction.

## Further Notes

Agentation research snapshot:

- Repo cloned locally under `.pi/research/agentation` at commit `8158a97c10c37e577b0a6e2d3175d143918216cd`.
- Agentation identifies elements with a utility that builds a readable element path, handles Shadow DOM parent traversal, recognizes buttons/links/inputs/headings/text/media/containers, and extracts nearby text. Relevant file: `package/src/utils/element-identification.ts`.
- Agentation's toolbar uses `document.elementFromPoint` plus Shadow DOM drilling to find the deepest element under the cursor. Relevant file: `package/src/components/page-toolbar-css/index.tsx`.
- Agentation handles hover by listening to `document` `mousemove`, ignoring its own toolbar, identifying the element, reading `getBoundingClientRect`, and storing hover info.
- Agentation handles click by listening on the document in capture phase, ignoring toolbar/popup/marker elements, preventing interactive defaults when annotation mode is active, reading the clicked element with coordinate lookup, capturing selected text via `window.getSelection`, capturing bounding box and nearby context, then opening a pending annotation popup.
- Agentation's annotation type includes many fields: element name/path, coordinates, selected text, bounding box, nearby text, CSS classes, computed styles, full path, accessibility, React components, and source file. UMKM Cepat MVP should use a smaller hidden subset.
- Agentation's popup pattern is a good UX reference: small focused textarea near the clicked element, cancel/add actions, keyboard submit, and focus management. UMKM Cepat should adapt the pattern visually to its warm dark workspace style and Indonesian copy.

Recommended implementation order:

1. Add annotation mode state to workspace and topbar activation.
2. Add annotation panel in the left workspace area.
3. Add private preview bridge injection and iframe postMessage contract.
4. Implement hover highlight and click target capture in the preview bridge.
5. Implement popup note entry and annotation list/pins.
6. Implement send-to-AI summary plus edit-route integration.
7. Add success clear/failure preserve behavior.
8. Add tests at workspace state, payload formatting, and edit submission seams.
